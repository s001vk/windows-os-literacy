Windows OS教養講座 第4章

ユーザー、権限、認証

ローカルユーザー、ドメインユーザー、SID、グループ、UAC、NTFS権限、共有権限、Active Directory、Entra IDの関係を学びます。

この章の目的

第3章では、Windows上のファイルやパスの見方を扱いました。 第4章では、そのファイルやサービス、共有フォルダに対して「誰が」「どの権限で」アクセスしているのかを理解します。

業務システムの障害では、ファイルや設定が存在していても、実行ユーザーに権限がないために失敗することがあります。 また、手動実行では成功するのに、サービス、IIS、タスクスケジューラでは失敗する場合、実行ユーザーの違いが原因になることがあります。

この章のゴール： Windowsの認証と権限を、ローカルPCだけでなく、ドメイン、共有フォルダ、業務アプリ実行ユーザーの観点で説明できるようになることです。

1. ユーザーの種類

Windowsにサインインするユーザーには、複数の種類があります。 どの種類のユーザーで動いているかにより、利用できるリソース、認証先、管理方法が変わります。

種類 説明 実務で見る場面
ローカルユーザー そのPCやサーバー内に作成されたユーザーです。 スタンドアロン端末、検証環境、ローカル管理者、サービス実行用アカウント。
ドメインユーザー Active Directoryドメインで管理されるユーザーです。 企業内のサインイン、ファイルサーバー、業務アプリ、シングルサインオン。
Microsoftアカウント 個人向けMicrosoftサービスと連携するアカウントです。 個人利用や一部小規模環境。企業運用では組織の方針に従います。
サービスアカウント サービスやタスクを実行するために使うアカウントです。 Windowsサービス、IIS、バッチ、スケジュール処理、監視エージェント。
基本： Windowsでは「人がログオンしているユーザー」と「プログラムが実行されているユーザー」が同じとは限りません。

2. AdministratorとUAC

Administratorは、Windowsで強い管理権限を持つ管理者アカウントや管理者グループを指す文脈で使われます。 管理者権限があれば、ソフトウェアのインストール、サービス操作、システム設定変更、他ユーザー領域へのアクセスなどが可能になります。

UACは、User Account Controlの略で、管理者権限が必要な操作を行う際に確認や昇格を求める仕組みです。 管理者グループに所属していても、常にすべての操作が管理者権限で実行されるわけではありません。

1. 標準権限で起動
通常のアプリやコマンドは、不要な管理権限を持たずに起動します。
2. 管理操作を要求
システム設定変更、Program Files配下への書き込み、サービス操作などで昇格が必要になります。
3. UACで確認
管理者の承認や資格情報入力を求めます。
4. 昇格後に実行
許可された操作だけを管理者権限で実行します。
安全な運用： UACを無効化するのではなく、必要な作業だけ管理者として実行し、通常業務は最小権限で行うことが基本です。

3. SIDとグループ

SIDはSecurity Identifierの略で、Windowsがユーザーやグループを識別するための内部IDです。 表示名が変わっても、権限判定ではSIDが重要になります。

グループは、複数のユーザーにまとめて権限を付与するための仕組みです。 企業運用では、個別ユーザーに直接権限を付けるより、グループに権限を付け、ユーザーをグループへ所属させる設計が一般的です。

ユーザー山田、svc_batch、appuser
グループ経理部、AppOperators、Administrators
SID内部的な識別子
権限ファイル、共有、サービス、ログ

障害調査では、ユーザー名だけでなく、所属グループ、実際に付与されている権限、拒否権限の有無を確認します。

4. 認証の考え方

認証は、利用者やシステムが「誰であるか」を確認する処理です。 Windowsでは、パスワード、Windows Hello、スマートカード、Kerberos、NTLM、クラウド認証など、環境により複数の方式が使われます。

Windows Helloは、生体認証やPINを使って端末へのサインインを安全かつ扱いやすくする仕組みです。 企業環境では、Active DirectoryやEntra ID、端末管理ポリシーと組み合わせて利用されることがあります。

用語 意味 実務での注意
認証 誰であるかを確認する。 パスワード、証明書、Windows Hello、Kerberosなど。
認可 認証された主体に、何を許可するかを決める。 ファイル権限、共有権限、管理者権限、アプリ権限。
監査 誰が何をしたかを記録する。 Securityログ、アプリ監査ログ、アクセスログ。
区別： ログインできることは認証の成功です。ファイルを読めることやサービスを操作できることは認可の結果です。

5. NTFS権限と共有権限

ファイルやフォルダに対するアクセスは、NTFS権限で制御されます。 ネットワーク共有経由でアクセスする場合は、共有権限も関係します。

権限 対象 代表的な確認事項
NTFS権限 ファイルやフォルダそのもの 読み取り、書き込み、実行、変更、所有者、継承、拒否権限。
共有権限 ネットワーク共有として公開された入口 共有名、許可ユーザー、読み取り/変更、アクセス経路。

実効権限は、複数の許可と拒否、グループ所属、継承、共有権限、NTFS権限の組み合わせで決まります。 そのため、単にフォルダのプロパティだけを見るのではなく、実行ユーザーで実際にアクセスできるかを確認します。

設計上の注意： 共有フォルダやアプリフォルダの権限は、個人名ではなく業務ロールやサービス用グループに付与すると、退職や異動、運用引き継ぎに強くなります。

6. Active DirectoryとEntra ID

Active Directoryは、企業内のユーザー、コンピューター、グループ、ポリシーを管理する基盤です。 ドメイン参加したWindows端末は、ドメインユーザーでサインインし、ファイルサーバーや業務アプリに対して統合的に認証できます。

Entra IDは、MicrosoftのクラウドID基盤です。 Microsoft 365やクラウドアプリ、デバイス管理と関係します。 企業環境では、オンプレミスのActive DirectoryとEntra IDを連携している場合もあります。

基盤 主な領域 業務での見方
Active Directory 社内ドメイン、端末、ユーザー、グループ、グループポリシー ファイルサーバー、Windows認証、端末管理、社内SSO。
Entra ID クラウドID、Microsoft 365、SaaS、条件付きアクセス クラウドアプリ認証、端末登録、多要素認証、ゼロトラスト設計。

どちらを使うか、または連携しているかは企業の設計により異なります。 認証障害を調査する場合は、端末がローカル管理、ドメイン参加、クラウド参加、ハイブリッド構成のどれに近いかを確認します。

7. 「ファイルがない」ではなく「権限がない」

Windowsでは、権限がないためにファイルやフォルダへアクセスできない場合があります。 エラーメッセージ上は「見つからない」「アクセスできない」「パスが無効」と表示されることがあり、原因の見分けが必要です。

1. パス確認
入力されたパス、ドライブ、UNCパス、相対パスの基準を確認します。
2. 実行ユーザー確認
誰の権限で処理しているかを確認します。サービスやIISでは人のログオンユーザーと異なります。
3. 権限確認
NTFS権限、共有権限、グループ所属、拒否権限を確認します。
4. ログ確認
アプリログ、イベントログ、Securityログ、ファイルサーバー側ログを確認します。
調査の型： 「存在確認」と「実行ユーザーでのアクセス確認」を分けるだけで、権限起因の問題を見つけやすくなります。

8. 実務での観点

ユーザー、権限、認証は、開発・運用の両方で重要です。 特に業務アプリをWindowsサービスやIISで動かす場合、開発者がログオンして実行した結果と、サービスアカウントで実行した結果は一致しないことがあります。

  • サービス実行ユーザーに、必要なフォルダ、共有、証明書秘密鍵、DB接続に対する権限があるか確認します。
  • 共有フォルダは、共有権限とNTFS権限を合わせて確認します。
  • 管理者権限でしか動かないアプリは、必要権限の設計が不十分な可能性があります。
  • ドメイン環境では、ユーザーの所属グループ、グループポリシー、認証先の状態も確認します。
  • 権限を広げる場合は、暫定対応か恒久対応かを区別し、作業記録を残します。

9. コマンド例

認証と権限の調査では、現在のユーザー、所属グループ、権限設定を確認します。

目的 コマンド例 確認できること
現在のユーザー whoami 現在の実行ユーザーを確認できます。
所属グループ whoami /groups ユーザーが所属するグループやSIDを確認できます。
ローカルユーザー Get-LocalUser ローカルユーザー一覧を確認できます。
権限確認 icacls "D:\app\data" 対象フォルダのNTFS権限を確認できます。
ポリシー結果 gpresult /r 適用されているグループポリシーの概要を確認できます。 
whoami
whoami /groups
icacls "\\fileserver\share\reports"
runas /user:DOMAIN\appuser cmd

runas で別ユーザーとして確認する場合は、運用ルールに従い、共有アカウントやパスワードの扱いに注意します。

第4章のまとめ

  1. Windowsには、ローカルユーザー、ドメインユーザー、Microsoftアカウント、サービスアカウントなどがある。
  2. 管理者グループに所属していても、UACにより通常は標準権限で実行されることがある。
  3. SIDは、Windowsがユーザーやグループを内部的に識別するためのIDである。
  4. 権限は個別ユーザーではなく、グループに付与すると運用しやすい。
  5. 認証は誰であるかの確認、認可は何を許可するかの判断である。
  6. 共有フォルダでは、共有権限とNTFS権限の両方が関係する。
  7. Active DirectoryとEntra IDは、企業Windowsの認証・端末管理・クラウド連携に関係する。
  8. 「ファイルがない」と見える問題でも、実際は実行ユーザーの権限不足である場合がある。

10. 理解度チェック

Q1. ローカルユーザーとドメインユーザーの違いは何ですか。

回答例を見る

ローカルユーザーはそのPCやサーバー内で管理されるユーザーです。 ドメインユーザーはActive Directoryドメインで管理され、複数の端末やサーバー、共有リソースで利用されます。

Q2. UACは何のための仕組みですか。

回答例を見る

管理者権限が必要な操作を行う際に確認や昇格を求め、通常時は不要な管理権限で動かないようにするための仕組みです。

Q3. SIDとは何ですか。

回答例を見る

Security Identifierの略で、Windowsがユーザーやグループを内部的に識別するためのIDです。 表示名ではなくSIDに基づいて権限判定されます。

Q4. 共有フォルダにアクセスできない場合、どの権限を確認しますか。

回答例を見る

共有権限とNTFS権限の両方を確認します。 さらに、実行ユーザー、資格情報、グループ所属、ネットワーク到達性、名前解決も確認します。

Q5. 手動実行では成功する処理がWindowsサービスでは失敗する場合、何を疑いますか。

回答例を見る

サービスの実行ユーザー、NTFS権限、共有権限、AppDataや環境変数の違い、資格情報、作業フォルダの違いを疑います。
