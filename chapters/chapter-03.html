Windows OS教養講座 第3章

ファイルシステムとパス

ドライブレター、絶対パス、Program Files、AppData、System32、NTFS、共有フォルダ、UNCパスを、業務アプリ運用の視点で学びます。

この章の目的

第2章では、Windows上でアプリが動くときにプロセス、サービス、DLL、設定、ログが関係することを扱いました。 第3章では、それらが実際に置かれる場所と、パスの考え方を整理します。

業務システムでは、設定ファイル、ログ、アップロードファイル、バッチ、証明書、共有フォルダ、ランタイムの配置が動作に直結します。 パスの解釈を誤ると、開発端末では動くが本番端末では動かない、サービス実行時だけ失敗する、共有フォルダにアクセスできない、といった問題につながります。

この章のゴール： Windowsの代表的なフォルダ、パス表記、NTFS権限、共有パスの意味を理解し、調査時に見るべき場所を判断できるようになることです。

1. ドライブレターとパス

Windowsでは、ストレージや一部の割り当て先を C: や D: のようなドライブレターで表します。 一般的にOSは C: に配置されますが、業務環境ではデータ領域、ログ領域、アプリ領域が別ドライブになっていることがあります。

パスは、ファイルやフォルダの場所を示す文字列です。 Windowsでは C:\Windows\System32 のようにバックスラッシュで階層を区切る表記が多く使われます。 PowerShellではスラッシュを受け付ける場面もありますが、Windowsの設定画面や古いアプリではバックスラッシュ前提のことがあります。

C:OS、Program Files、Users
D:業務データ、ログ、アプリ配置
\\server\share共有フォルダ、ファイルサーバー
環境変数%TEMP%、%APPDATA%、PATH
注意： 開発端末では C: にある前提のパスでも、本番サーバーでは別ドライブや共有パスになっていることがあります。絶対パスの固定は運用差異を生みやすい点に注意します。

2. 絶対パスと相対パス

絶対パスは、ドライブや共有名から始まる完全な場所指定です。 相対パスは、現在の作業フォルダを基準にした場所指定です。

種類 例 特徴 実務での注意
絶対パス C:\app\config\appsettings.json どこから実行しても同じ場所を指す。 環境差異に弱い。ドライブや配置先が変わると修正が必要。
相対パス .\config\appsettings.json 現在の作業フォルダを基準に解釈される。 サービス、タスク、IISでは作業フォルダが想定と異なる場合がある。
UNCパス \\fileserver\share\report.csv ネットワーク共有上の場所を示す。 認証、共有権限、NTFS権限、ネットワーク到達性が関係する。

サービスやタスクスケジューラで相対パスを使う場合、手動実行時とは基準フォルダが違うことがあります。 設定ファイルやログ出力先は、実行方式ごとの作業フォルダを確認して設計します。

3. 代表的なWindowsフォルダ

Windowsには、用途が分かれた標準フォルダがあります。 業務アプリの配置や調査では、それぞれの意味を理解しておく必要があります。

フォルダ 主な用途 調査時の見方
C:\Program Files 64bitアプリケーションの標準的なインストール先 手動書き込みは制限されることが多い。設定やログの置き場に向かない場合がある。
C:\Program Files (x86) 32bitアプリケーションの標準的なインストール先 32bit / 64bitの依存関係を確認する。
C:\ProgramData 全ユーザー共通のアプリ設定やデータ サービスや複数ユーザーで共有する設定の置き場所になりやすい。
C:\Users\ユーザー名\AppData ユーザーごとのアプリデータ ログオンユーザーごとに場所が異なる。サービス実行ユーザーでは想定と違う場合がある。
C:\Windows\System32 Windowsのシステムファイル、64bit向けシステムコンポーネント 直接変更しない。OS機能やコマンドの実体が含まれる。
C:\Windows\SysWOW64 64bit Windows上の32bit互換用システムコンポーネント 名前と実体の関係が直感と逆に見えるため、bit数差異の調査で注意する。
実務観点： アプリ本体はProgram Files、全体設定はProgramData、ユーザー別設定はAppData、ログは専用フォルダや標準ログ領域、というように用途で分けると運用しやすくなります。

4. NTFSとファイル権限

NTFSは、Windowsで広く使われるファイルシステムです。 ファイルやフォルダに対して、読み取り、書き込み、実行、変更、フルコントロールなどの権限を設定できます。

業務システムでは、ファイルが存在していても実行ユーザーに権限がなければアクセスできません。 そのため、障害調査では「ファイルがあるか」と「そのユーザーが読めるか、書けるか」を分けて確認します。

1. ファイルの存在
パスが正しいか、ドライブや共有が存在するかを確認します。
2. 実行ユーザー
ログオンユーザー、サービスアカウント、IISアプリケーションプールIDなどを確認します。
3. NTFS権限
対象フォルダに読み取り、書き込み、実行権限があるかを確認します。
4. ロックや容量
別プロセスのロック、ディスク容量不足、パス長、ファイル名の制約を確認します。
注意： 調査のために安易にEveryoneフルコントロールを付与すると、情報漏えいや改ざんのリスクが増えます。必要なユーザーやサービスアカウントに必要な権限だけを付与します。

5. 共有フォルダとUNCパス

Windowsでは、ファイルサーバー上のフォルダを共有し、クライアントやサーバーからアクセスできます。 共有先は \\サーバー名\共有名\フォルダ のようなUNCパスで表します。

共有フォルダには、共有権限とNTFS権限の両方が関係します。 共有権限で許可されていてもNTFS権限で拒否されればアクセスできません。 逆にNTFS権限があっても共有権限で制限される場合があります。

観点 確認内容
名前解決 サーバー名がDNSやNetBIOSで解決できるか。
ネットワーク SMB通信に必要な経路やFirewallが許可されているか。
認証 アクセスしているユーザーが意図した資格情報か。
権限 共有権限とNTFS権限の両方で許可されているか。

6. ログや設定ファイルの置き場所

業務アプリでは、設定ファイルとログファイルの置き場所を明確にすることが重要です。 アプリ本体のフォルダにログを書き込む設計は、Program Files配下では権限不足になりやすく、運用にも向かない場合があります。

  • 全ユーザー共通の設定は ProgramData 配下に置くことがあります。
  • ユーザーごとの設定は AppData 配下に置くことがあります。
  • サービスのログは、専用のログフォルダやイベントログに出す設計が扱いやすくなります。
  • IISのログ、アプリログ、Windowsイベントログは、それぞれ保存場所と保持期間を確認します。
運用設計： ログ出力先は、容量監視、ローテーション、アクセス権、障害時の収集方法まで含めて決める必要があります。

7. 32bit / 64bit差異

64bit Windowsでは、64bitアプリと32bitアプリが共存できます。 しかし、参照するフォルダやレジストリ、DLL探索が異なるため、互換性問題の原因になります。

項目 64bit側 32bit側
アプリ配置 Program Files Program Files (x86)
システムフォルダ System32 SysWOW64
主な問題 64bit DLLが必要なアプリで32bit DLLを参照する。 32bitアプリが64bit専用コンポーネントを利用できない。

フォルダ名だけで判断すると誤解しやすいため、実行中プロセスのbit数、利用するDLL、ランタイム、ODBCドライバなどを合わせて確認します。

8. 実務での観点

ファイルシステムとパスの知識は、アプリ配置、ログ収集、権限設計、バックアップ、移行、障害調査に直結します。 特にサービスやIISのように利用者の画面操作とは異なる実行ユーザーで動く処理では、手動確認だけで判断しないことが重要です。

  • 「手動では読める」が「サービスでは読めない」場合、実行ユーザーとNTFS権限を確認します。
  • 「共有フォルダに書けない」場合、UNCパス、資格情報、共有権限、NTFS権限、ネットワークを分けて確認します。
  • 「本番だけDLLが見つからない」場合、Program Files、System32、SysWOW64、PATH、ランタイムの差異を確認します。
  • 「ログが出ない」場合、出力先フォルダの存在、書き込み権限、容量、ログ設定を確認します。

9. コマンド例

パス、存在確認、権限、共有状態は、コマンドで確認して記録すると調査の再現性が高まります。

目的 コマンド例 確認できること
ファイル存在確認 Test-Path "C:\app\config.json" 指定パスが存在するかを確認できます。
フォルダ一覧 Get-ChildItem "C:\ProgramData" フォルダ内のファイルやサブフォルダを確認できます。
権限確認 icacls "D:\logs" NTFS権限の設定を確認できます。
共有接続確認 net use 現在のネットワーク共有接続を確認できます。
環境変数確認 $env:PATH DLLやコマンド探索に影響するPATHを確認できます。 
Test-Path "\\fileserver\share\input.csv"
icacls "D:\app\logs"
Get-ChildItem Env:
Resolve-Path ".\config\appsettings.json"

第3章のまとめ

  1. Windowsでは、ドライブレター、絶対パス、相対パス、UNCパスを使い分ける。
  2. 相対パスは現在の作業フォルダに依存するため、サービスやタスク実行では注意が必要である。
  3. Program Files、ProgramData、AppData、System32、SysWOW64は用途が異なる。
  4. NTFS権限により、ファイルが存在しても実行ユーザーがアクセスできない場合がある。
  5. 共有フォルダでは、名前解決、ネットワーク、認証、共有権限、NTFS権限を分けて確認する。
  6. ログや設定ファイルは、権限、容量監視、収集方法を含めて配置を考える。
  7. 32bit / 64bit差異は、フォルダ、DLL、ランタイム、レジストリ参照に影響する。

10. 理解度チェック

Q1. 絶対パスと相対パスの違いは何ですか。

回答例を見る

絶対パスはドライブや共有名から始まる完全な場所指定です。 相対パスは現在の作業フォルダを基準に解釈されるため、実行方法によって指す場所が変わる場合があります。

Q2. Program Files配下にログを書き込む設計で起きやすい問題は何ですか。

回答例を見る

標準ユーザーやサービス実行ユーザーに書き込み権限がなく、ログ出力に失敗する場合があります。 また、アプリ本体と運用データが混在し、バックアップや容量管理もしづらくなります。

Q3. 「ファイルがない」と見えるが実際は権限不足のケースでは、何を確認しますか。

回答例を見る

実行ユーザー、対象パス、NTFS権限、共有権限、アクセス時の資格情報、イベントログやアプリログを確認します。

Q4. UNCパスとは何ですか。

回答例を見る

\\server\share\folder のように、ネットワーク共有上の場所を示すパスです。 アクセスには名前解決、ネットワーク到達性、認証、共有権限、NTFS権限が関係します。

Q5. 32bit / 64bit差異で注意すべき代表的なフォルダは何ですか。

回答例を見る

Program Files、Program Files (x86)、System32、SysWOW64 です。 DLLやランタイムのbit数も合わせて確認します。
