Windows OS教養講座 第6章

ネットワーク基礎

IPアドレス、サブネット、DNS、DHCP、hosts、プロキシ、VPN、RDP、SMB、WinRM、Windows Firewall、ポート番号を実務調査の観点で学びます。

この章の目的

第5章では、Windows上で動くプロセスやサービスを扱いました。 第6章では、それらの処理が社内システム、ファイルサーバー、Web API、DB、クラウド、利用者端末と通信するための基礎を整理します。

業務システムでは、「サーバーに接続できない」「社内では動くが在宅では動かない」「名前解決できない」「ポートが閉じている」「プロキシ経由で失敗する」といった問題が頻繁に起きます。 ネットワークを層で見られると、原因を速く切り分けられます。

この章のゴール： Windows環境の通信トラブルを、IP、名前解決、経路、ポート、Firewall、プロキシ、VPN、認証の観点で分類できるようになることです。

1. IPアドレス、サブネット、デフォルトゲートウェイ

IPアドレスは、ネットワーク上の端末やサーバーを識別するための住所に相当します。 サブネットは、同じネットワーク範囲に属する端末を区切る考え方です。 デフォルトゲートウェイは、自分のネットワーク外へ通信するときに最初に渡すルーターです。

項目 役割 問題時の見方
IPアドレス 端末やサーバーのネットワーク上の識別子 想定したネットワーク帯か、重複していないかを確認する。
サブネットマスク 同一ネットワーク範囲を判断するための情報 サブネット設定の誤りで近い宛先に届かない場合がある。
デフォルトゲートウェイ 別ネットワークへ出る入口 未設定や誤設定では、外部や別セグメントへ到達できない。
PC192.168.10.25
同一サブネット192.168.10.0/24
ゲートウェイ192.168.10.1
別ネットワーク社内サーバー、インターネット

業務環境では、端末の有線LAN、無線LAN、VPN、仮想アダプタが同時に存在することがあります。 そのため、どのインターフェースで通信しようとしているかも確認します。

2. DNS、DHCP、hosts

DNSは、ホスト名やドメイン名をIPアドレスへ変換する仕組みです。 業務アプリの接続先が api.example.local のような名前で設定されている場合、まず名前解決が成功する必要があります。

DHCPは、端末にIPアドレス、サブネットマスク、デフォルトゲートウェイ、DNSサーバーなどを自動配布する仕組みです。 hostsファイルは、端末ローカルで名前とIPアドレスの対応を設定するファイルです。

仕組み 役割 調査ポイント
DNS 名前をIPアドレスへ変換する。 参照しているDNSサーバー、検索サフィックス、内部名と外部名の違い。
DHCP ネットワーク設定を自動配布する。 リース取得状況、誤ったDNSやゲートウェイの配布、VPN接続時の差異。
hosts 端末ごとに名前解決を上書きする。 古い検証設定が残っていないか、意図しないIPへ向いていないか。
注意： hostsファイルでの暫定回避は、端末ごとの設定差異を生みます。恒久対応ではDNSや環境設定を正しく直すことを優先します。

3. プロキシとVPN

プロキシは、端末やアプリの外部通信を中継する仕組みです。 企業では、Webアクセスの制御、ログ取得、セキュリティ検査、認証のためにプロキシを利用することがあります。

VPNは、社外や別拠点から社内ネットワークへ安全に接続するための仕組みです。 VPN接続時は、IPアドレス、DNS、ルーティング、プロキシ設定が社内接続時と変わることがあります。

項目 関係する問題 確認事項
プロキシ Web API接続失敗、認証エラー、証明書エラー、タイムアウト OS設定、ブラウザ設定、WinHTTP設定、アプリ独自設定、認証方式。
VPN 社内名の解決不可、ファイルサーバー不可、経路不一致 VPN接続状態、割り当てIP、DNS、ルート、分割トンネル設定。
実務観点： ブラウザでは接続できるがサービスでは接続できない場合、ブラウザのプロキシ設定とWindowsサービスが使うWinHTTP設定が異なることがあります。

4. RDP、SMB、WinRM

Windows運用でよく使う通信には、RDP、SMB、WinRMがあります。 それぞれ用途、ポート、認証、Firewall設定が異なります。

名称 用途 代表的なポート 注意点
RDP リモートデスクトップ接続 3389 公開範囲、認証、監査、接続元制限を適切に管理する。
SMB ファイル共有、プリンタ共有 445 共有権限、NTFS権限、Firewall、古いプロトコル利用に注意する。
WinRM PowerShellリモート管理 5985 / 5986 HTTPS利用、認証、許可ホスト、管理対象範囲を設計する。
安全な運用： RDPやWinRMを広い範囲へ開放するのは避け、VPN、接続元制限、多要素認証、監査ログ、踏み台設計などを組み合わせます。

5. Windows Firewallとポート番号

Windows Firewallは、端末やサーバーへの受信通信、または環境によっては送信通信を制御する仕組みです。 ネットワークプロファイルには、ドメイン、プライベート、パブリックがあり、接続しているネットワークによって適用ルールが変わる場合があります。

ポート番号は、同じIPアドレス上で複数のサービスを区別するために使われます。 たとえばHTTPは80、HTTPSは443、RDPは3389、SMBは445が代表例です。

クライアント接続元IP、アプリ
ネットワーク経路ルーター、VPN、Proxy
Windows Firewallプロファイル、受信規則
サービスIP:Portで待ち受け

通信できないときは、アプリが待ち受けているか、ポートが開いているか、Firewallが許可しているか、途中経路で遮断されていないかを分けて確認します。

6. 通信が成立するまでの流れ

Windowsでアプリが通信するときは、概念的に次の順で条件が満たされる必要があります。 障害調査では、上から順に確認すると原因候補を整理しやすくなります。

1. 接続先の指定
URL、ホスト名、IPアドレス、ポート番号、プロトコルが正しいかを確認します。
2. 名前解決
DNSまたはhostsにより、ホスト名がIPアドレスへ解決できるかを確認します。
3. 経路
同一サブネット、ゲートウェイ、VPN、ルーティングを通じて到達できるかを確認します。
4. ポート到達
接続先サービスが待ち受けており、Firewallや経路上の制御で遮断されていないかを確認します。
5. プロキシや認証
必要なプロキシ、資格情報、Windows認証、証明書が正しく使われているかを確認します。
6. アプリケーション応答
HTTPステータス、アプリログ、サーバーログ、イベントログで処理結果を確認します。

7. 社内では動くが在宅では動かない原因分類

在宅勤務や社外ネットワークでは、社内LANとは前提が変わります。 同じPCでも、VPN有無、DNS、ルート、プロキシ、Firewall、端末管理ポリシー、証明書利用が変わる場合があります。

原因分類 典型例 確認方法
名前解決 社内DNSでしか解決できない名前をVPNなしで使っている。 nslookup、DNSサーバー、検索サフィックス。
経路 VPN接続時だけ社内セグメントへのルートが追加される。 route print、VPN接続状態、割り当てIP。
プロキシ 社外ではプロキシ経由が必要、または社内プロキシに届かない。 Windows設定、ブラウザ設定、WinHTTP設定、アプリ設定。
ポート制御 自宅回線やVPNポリシーで特定ポートが許可されていない。 Test-NetConnection、Firewall、VPNポリシー。
認証・証明書 社内端末証明書やWindows認証の前提が満たされない。 証明書ストア、資格情報、ドメイン到達性、イベントログ。

8. 実務での観点

ネットワーク調査では、「つながらない」を一つの問題として扱わず、名前解決、経路、ポート、認証、アプリ応答に分けます。 Windowsではさらに、Firewallプロファイル、プロキシ設定、VPNアダプタ、ドメイン参加状態、資格情報キャッシュが関係することがあります。

  • 接続先が名前かIPかを確認し、名前解決とIP到達性を分けて調べます。
  • ポートが開いているかを確認し、アプリが待ち受けているかとFirewall許可を分けます。
  • ブラウザとサービスで挙動が違う場合、プロキシ設定と実行ユーザーの違いを確認します。
  • VPN接続時は、DNSサーバー、ルート、割り当てIP、分割トンネルの有無を確認します。
  • RDP、SMB、WinRMは便利ですが、公開範囲、認証、監査、接続元制限を設計して使います。
調査の型： 「名前が引けるか」「IPに届くか」「ポートが開いているか」「認証できるか」「アプリが返しているか」を分けると、曖昧な通信障害を扱いやすくなります。

9. コマンド例

Windowsのネットワーク状態は、標準コマンドとPowerShellで確認できます。

目的 コマンド例 確認できること
IP設定 ipconfig /all IP、DNS、DHCP、ゲートウェイ、アダプタ情報を確認できます。
名前解決 nslookup server.example.local 名前がどのDNSでどのIPへ解決されるかを確認できます。
到達確認 ping 192.168.10.10 ICMP応答の有無を確認できます。ただし応答禁止の環境もあります。
経路確認 tracert server.example.local 接続先までの経路を確認できます。
ポート確認 Test-NetConnection server.example.local -Port 443 指定ポートへTCP接続できるかを確認できます。
WinHTTPプロキシ netsh winhttp show proxy サービスや一部アプリが参照するWinHTTPプロキシ設定を確認できます。
Firewallプロファイル Get-NetFirewallProfile ドメイン、プライベート、パブリックのFirewall状態を確認できます。 
ipconfig /all
nslookup intranet.example.local
Test-NetConnection intranet.example.local -Port 443
Get-NetIPConfiguration
Get-NetTCPConnection -State Listen
netsh winhttp show proxy
解釈の注意： ping が失敗しても、必ずしもアプリ通信が失敗するとは限りません。ICMPが遮断されているだけの場合があります。業務アプリのポートで確認します。

第6章のまとめ

  1. IPアドレス、サブネット、デフォルトゲートウェイは、端末がどのネットワークへ通信できるかを決める基礎である。
  2. DNSは名前解決、DHCPはネットワーク設定の自動配布、hostsは端末ローカルの名前解決上書きである。
  3. プロキシとVPNは、通信経路、認証、名前解決、ルーティングに影響する。
  4. RDP、SMB、WinRMはWindows運用で重要だが、公開範囲と認証を慎重に設計する。
  5. Windows Firewallは、ネットワークプロファイルや規則により受信・送信通信を制御する。
  6. 通信調査では、名前解決、経路、ポート、プロキシ、認証、アプリ応答を分けて確認する。
  7. 社内では動くが在宅では動かない場合、VPN、DNS、ルート、プロキシ、証明書、Firewallの差異を確認する。

10. 理解度チェック

Q1. デフォルトゲートウェイは何のために使われますか。

回答例を見る

自分のネットワーク外へ通信するときに最初に渡すルーターです。 未設定や誤設定では、別セグメントや外部ネットワークへ到達できない場合があります。

Q2. DNSとhostsファイルの違いは何ですか。

回答例を見る

DNSはネットワーク上の名前解決基盤です。 hostsファイルは端末ローカルで名前とIPアドレスの対応を上書きするファイルです。

Q3. ブラウザでは接続できるがWindowsサービスでは接続できない場合、何を疑いますか。

回答例を見る

プロキシ設定の違い、WinHTTP設定、サービス実行ユーザーの権限や資格情報、証明書ストア、Firewall、DNS設定の違いを疑います。

Q4. ポート番号を確認する理由は何ですか。

回答例を見る

接続先サービスがどの入口で待ち受けているか、Firewallや経路で遮断されていないか、別プロセスと競合していないかを確認するためです。

Q5. 社内では動くが在宅では動かない場合、代表的な原因分類を挙げてください。

回答例を見る

VPN未接続または経路不足、社内DNSに到達できない、プロキシ設定差異、Firewallやポート制御、証明書やWindows認証の前提不足などが考えられます。
