Windows OS教養講座 第2章

Windowsの基本構造

Windows内部で、アプリケーション、プロセス、スレッド、サービス、ドライバ、DLL、レジストリ、ログがどのように関係しているかを学びます。

この章の目的

第1章では、Windowsを企業システムの基盤として見るための全体像を扱いました。 第2章では、その基盤の内部構造を、障害調査や運用で使える粒度に分解します。

業務システムでは、「アプリが起動しない」「サービスが落ちる」「通信できない」「ログが出ない」といった問題を扱います。 これらはアプリケーションだけでなく、OSの実行管理、権限、設定、ドライバ、イベントログと結び付いています。

この章のゴール： Windows上でアプリケーションが動くとき、裏側でどの部品が関係しているかを説明できるようになることです。

1. カーネルの役割

カーネルは、OSの中核です。 CPU、メモリ、ディスク、ネットワーク、プロセス、スレッド、デバイスなどを管理し、アプリケーションが直接ハードウェアを扱わなくてもよいようにします。

たとえば業務アプリがファイルを読み込むとき、アプリはディスク装置を直接制御しているわけではありません。 WindowsのAPIを呼び出し、その要求がファイルシステムやデバイスドライバを通じて処理されます。

業務アプリEXE、DLL、.NET、Javaなど
Windows APIファイル、通信、画面、認証の呼び出し口
カーネルCPU、メモリ、I/O、プロセスを管理
ハードウェアディスク、NIC、USB、画面
重要： カーネルは見えにくい存在ですが、アプリケーションの実行、通信、ファイルアクセス、デバイス利用の土台です。

2. ユーザーモードとカーネルモード

Windowsでは、アプリケーションが動く領域と、OSの中核が動く領域を分けています。 代表的にはユーザーモードとカーネルモードです。

領域 主な実行対象 特徴 障害時の見方
ユーザーモード 通常のアプリケーション、ユーザーサービスの一部 直接ハードウェアに触れない。異常終了してもOS全体への影響を抑えやすい。 アプリクラッシュ、例外、権限不足、依存DLL不足を疑う。
カーネルモード カーネル、デバイスドライバ、低レベルI/O 強い権限で動く。問題が起きるとOS全体に影響する場合がある。 ドライバ不具合、ハードウェア、OS更新後の互換性を疑う。

この分離により、通常アプリの不具合がすぐにOS全体の停止へつながらないよう設計されています。 ただし、デバイスドライバやセキュリティ製品のようにカーネルに近い領域で動くものは、影響範囲が大きくなりやすい点に注意が必要です。

3. プロセスとスレッド

プロセスは、実行中のプログラムの単位です。 メモリ空間、実行ファイル、開いているファイル、環境変数、権限情報などを持ちます。 タスクマネージャーで見えるアプリやバックグラウンド処理の多くは、プロセスとして確認できます。

スレッドは、プロセス内で実際に処理を進める実行単位です。 1つのプロセスが複数のスレッドを持つことで、画面描画、通信待ち、計算処理、ファイル読み書きを並行して扱えます。

プロセスメモリ空間、権限、環境変数を持つ
スレッドA画面処理
スレッドB通信処理
スレッドCファイル処理
障害調査の注意： 「アプリが重い」という症状でも、CPUを使っているプロセス、待機しているスレッド、ディスクI/O、ネットワーク待ちのどれが原因かで調査方法が変わります。

4. サービスとドライバ

Windowsサービスは、ユーザーが画面で操作していなくてもバックグラウンドで動くプログラムです。 データベース、監視エージェント、ウイルス対策、更新管理、ジョブ実行、Webアプリの補助処理などで使われます。

ドライバは、OSとハードウェアや低レベル機能をつなぐソフトウェアです。 プリンタ、ネットワークアダプタ、ストレージ、セキュリティ製品、仮想化製品などで重要です。

要素 役割 実務で見る場面
サービス 常駐処理、起動時実行、管理者が状態を制御する処理 起動失敗、停止、実行ユーザー、依存サービス、ログ確認
ドライバ デバイスや低レベル機能をOSから使えるようにする 周辺機器不具合、通信断、OS更新後の問題、ブルースクリーン

5. DLLと共有ライブラリ

DLLは、Windowsで使われる共有ライブラリです。 複数のアプリケーションが共通の処理を再利用できるようにする仕組みで、画面表示、暗号化、通信、ランタイム、業務アプリ固有の部品などに使われます。

DLLが見つからない、バージョンが違う、32bit版と64bit版を取り違える、といった問題は、Windowsアプリの典型的な実行トラブルです。 特に開発端末では動くが本番端末では動かない場合、依存DLLやランタイムの差異を確認します。

見るべき観点： EXE単体だけでアプリが動くとは限りません。DLL、.NET、Visual C++ Redistributable、Java Runtime、環境変数、PATHなどの依存関係も実行条件です。

6. レジストリ、ファイルシステム、イベントログ

Windowsの実行環境は、プログラム本体だけでは決まりません。 設定、ファイル配置、ログ出力先、ユーザー権限、OSポリシーが組み合わさって動作が決まります。

基盤 役割 調査で見るポイント
レジストリ OSやアプリの構成情報を階層的に保持する設定データベース 設定値、ポリシー、インストール情報、起動設定
ファイルシステム 実行ファイル、設定ファイル、ログ、データを保存する 存在確認、パス、NTFS権限、容量、ロック
イベントログ OS、サービス、セキュリティ、アプリの重要な事象を記録する エラー、警告、イベントID、発生時刻、サービス起動失敗

これらは第3章以降で詳しく扱います。 この章では、アプリの実行は「EXEを起動するだけ」ではなく、設定とログの基盤に支えられている点を押さえてください。

7. アプリが動くときに起きていること

Windowsでアプリケーションを起動すると、概念的には次のような流れが発生します。 実際の処理はアプリや環境により異なりますが、障害調査ではこの順に確認すると整理しやすくなります。

1. 実行ファイルを探す
ショートカット、PATH、サービス定義、タスク定義などからEXEやスクリプトを特定します。
2. 実行ユーザーを決める
ログオンユーザー、サービスアカウント、管理者権限の有無が決まります。
3. プロセスを作る
メモリ空間、環境変数、ハンドル、権限トークンを持つプロセスが生成されます。
4. DLLや設定を読み込む
依存DLL、設定ファイル、レジストリ、証明書、環境変数を参照します。
5. ファイルや通信を使う
データベース、共有フォルダ、HTTP API、DNS、ポート、Firewallが関係します。
6. ログを出す
アプリログ、イベントログ、標準出力、監視ログに結果やエラーが残ります。
運用での使い方： 起動できない場合は、実行ファイル、権限、依存DLL、設定、通信、ログの順に切り分けると、原因候補を狭めやすくなります。

8. 実務での観点

業務システムでは、Windowsの基本構造を知っていると、問題の層を分けて調査できます。 たとえば同じ「起動しない」でも、実行ファイルがない、DLLがない、権限がない、サービス実行ユーザーが違う、設定値が壊れている、OS更新後にドライバが合わないなど、原因は分かれます。

  • プロセスが存在するか、すぐ終了していないかを確認します。
  • サービスの場合は、起動種類、実行ユーザー、依存サービス、イベントログを確認します。
  • DLLやランタイム不足が疑われる場合は、導入済みコンポーネントとアプリのbit数を確認します。
  • 権限が関係する場合は、ファイル、レジストリ、サービス操作、ネットワーク共有へのアクセス権を分けて確認します。
  • OS更新やドライバ更新後の問題では、変更時刻とイベントログを合わせて確認します。
よくある誤解： 「アプリの問題」と決めつけると、サービス実行ユーザー、DLL依存、Firewall、証明書、イベントログのようなOS側の原因を見落とします。

9. コマンド例

Windowsの基本構造を確認するには、GUIだけでなくコマンドで状態を取得できると再現性が高まります。

目的 コマンド例 確認できること
プロセス一覧 Get-Process 実行中のプロセス名、CPU、メモリ使用状況を確認できます。
サービス一覧 Get-Service サービスの状態、名前、表示名を確認できます。
特定サービスの詳細 sc qc サービス名 起動種類、実行ファイルパス、依存関係などを確認できます。
イベントログ確認 Get-WinEvent -LogName System -MaxEvents 20 Systemログの直近イベントを確認できます。 
Get-Process | Sort-Object CPU -Descending | Select-Object -First 10
Get-Service | Where-Object Status -ne "Running"
Get-WinEvent -LogName Application -MaxEvents 20

これらの結果だけで原因が確定するとは限りません。 ただし、プロセス、サービス、ログの現在状態を揃えることで、調査の出発点を作れます。

第2章のまとめ

  1. カーネルは、CPU、メモリ、I/O、プロセス、デバイスを管理するOSの中核である。
  2. ユーザーモードとカーネルモードの分離により、通常アプリとOS中核の影響範囲を分けている。
  3. プロセスは実行中プログラムの単位で、スレッドはプロセス内で処理を進める単位である。
  4. Windowsサービスは、ログオンユーザーの画面操作とは独立して動く常駐処理である。
  5. ドライバは、OSとハードウェアや低レベル機能をつなぐため、障害時の影響が大きい場合がある。
  6. DLL、レジストリ、ファイルシステム、イベントログは、アプリ実行条件と調査材料になる。
  7. 「アプリが動く」とは、実行ファイル、権限、依存関係、設定、通信、ログが連携することである。

10. 理解度チェック

Q1. カーネルはWindowsの中でどのような役割を持ちますか。

回答例を見る

CPU、メモリ、ディスク、ネットワーク、プロセス、スレッド、デバイスなどを管理し、アプリケーションが安全にハードウェアやOS機能を利用できるようにします。

Q2. ユーザーモードとカーネルモードを分ける理由は何ですか。

回答例を見る

通常アプリケーションの不具合がOS全体に直接影響しにくくするためです。 カーネルモードは強い権限を持つため、ドライバ不具合などはOS全体に影響する場合があります。

Q3. プロセスとスレッドの違いは何ですか。

回答例を見る

プロセスは実行中プログラムの単位で、メモリ空間や権限情報を持ちます。 スレッドはプロセス内で実際に処理を進める実行単位です。

Q4. Windowsサービスの調査で確認すべき代表項目は何ですか。

回答例を見る

サービス状態、起動種類、実行ユーザー、実行ファイルパス、依存サービス、イベントログ、アプリケーションログを確認します。

Q5. アプリが起動しない場合、EXE以外に確認すべきものは何ですか。

回答例を見る

実行ユーザーの権限、依存DLL、ランタイム、設定ファイル、レジストリ、環境変数、通信先、証明書、イベントログなどを確認します。
