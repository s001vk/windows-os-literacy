Windows OS教養講座 第8章

イベントログとトラブルシューティング

イベントビューアー、Application、System、Security、Setup、Applications and Services Logs、イベントID、Get-WinEventを使い、障害時の事実を集める方法を学びます。

この章の目的

第7章では、Windowsの設定基盤であるレジストリを扱いました。 第8章では、Windowsやアプリケーションが記録するイベントログを使い、障害調査の初動で何を確認すべきかを整理します。

業務システムでは、利用者から「動かない」「遅い」「接続できない」と報告されても、原因がアプリ、OS、サービス、権限、ネットワーク、証明書、更新のどこにあるかはすぐには分かりません。 イベントログは、発生時刻と事象を結び付けるための重要な一次情報です。

この章のゴール： イベントログを、単なるエラー一覧ではなく、障害時に事実を集め、仮説を立て、切り分けるための材料として扱えるようになることです。

1. イベントログとは何か

イベントログは、Windows、サービス、ドライバ、セキュリティ機能、インストーラー、アプリケーションなどが記録する事象のログです。 ファイルベースのアプリログとは別に、Windows標準の仕組みとして管理されます。

イベントログには、発生時刻、ログ名、ソース、イベントID、レベル、メッセージ、コンピューター名、ユーザー情報などが含まれます。 これらを組み合わせることで、「いつ」「どのコンポーネントが」「どの種類の問題を記録したか」を確認できます。

発生時刻いつ起きたか
ソースどの機能が記録したか
イベントID事象の種類
メッセージ詳細と補足情報
基本： イベントログは「原因そのもの」ではなく、「原因を絞るための事実」です。ログの内容と再現条件、直前の変更、アプリログを合わせて読みます。

2. イベントビューアーの見方

イベントビューアーは、Windows標準のイベントログ確認ツールです。 GUIでログを検索、フィルター、保存できるため、初動調査や利用者端末での確認に向いています。

ただし、イベントビューアーで赤いエラーだけを眺めても、原因を正確に判断できるとは限りません。 重要なのは、障害が発生した時刻の前後に、関連するログがどの順番で出ているかを見ることです。

1. 発生時刻を決める
利用者報告、監視通知、アプリログ、リリース時刻から調査対象の時間帯を絞ります。
2. 関連ログを開く
Application、System、Security、対象アプリ固有ログを確認します。
3. レベルで絞る
エラーと警告を中心に見ますが、直前の情報ログも手がかりになります。
4. ソースとイベントIDを見る
同じイベントIDが繰り返されているか、別のソースと連動しているかを確認します。
5. アプリログと照合する
イベントログだけで判断せず、アプリケーションログや操作履歴と突き合わせます。
注意： Windowsには平常時でも警告やエラーが記録されることがあります。問題発生時刻と関係しない古いログを原因と決めつけないようにします。

3. 代表的なログの種類

イベントビューアーには複数のログがあります。 実務で最初に見ることが多いのは、Application、System、Security、Setupです。

ログ名 主な内容 よく見る場面
Application アプリケーション、.NET Runtime、Windows Error Reportingなどのログ。 アプリクラッシュ、例外、業務アプリ起動失敗、ランタイムエラー。
System OS、サービス、ドライバ、デバイス、ネットワーク関連のログ。 サービス起動失敗、ドライバ問題、ディスク、ネットワーク、OS側の異常。
Security ログオン、認証、監査、権限操作などのセキュリティログ。 ログオン失敗、権限変更、監査、認証トラブル。閲覧には権限が必要です。
Setup Windows Updateやインストール処理に関係するログ。 更新失敗、機能追加、インストーラー、再起動後の変化。
Applications and Services Logs Windows機能やアプリ固有の詳細ログ。 IIS、PowerShell、Task Scheduler、Defender、DNS Clientなどの詳細調査。
初動の型： アプリ問題はApplication、OSやサービス問題はSystem、認証や監査はSecurity、更新後の変化はSetupを起点にします。

4. エラー、警告、情報とイベントID

イベントログにはレベルがあります。 代表的には、エラー、警告、情報です。 エラーは失敗や停止を示すことが多く、警告は将来問題になる可能性や一時的な異常を示します。 情報は正常処理や状態変化を記録します。

レベル 意味 読み方
エラー 処理失敗、異常終了、起動失敗など。 障害時刻と一致するか、直前に関連ログがあるかを見る。
警告 失敗ではないが注意が必要な状態。 繰り返し発生しているか、後続のエラーにつながっているかを見る。
情報 正常な処理、開始、停止、状態変更。 処理の開始時刻や設定変更の痕跡として見る。

イベントIDは、同じソース内で事象の種類を識別する番号です。 ただし、イベントIDだけで意味が決まるわけではありません。 同じ番号でもソースが違えば意味が異なるため、ログ名、ソース、イベントID、メッセージをセットで確認します。

調査の注意： 「イベントIDだけで検索して解決策を適用する」のではなく、自分の環境、バージョン、直前の変更、発生条件に合っているか確認します。

5. クラッシュログとサービス起動失敗

アプリケーションがクラッシュした場合、ApplicationログにWindows Error Reporting、Application Error、.NET Runtimeなどのイベントが記録されることがあります。 例外コード、障害モジュール、プロセス名、発生時刻が手がかりになります。

Windowsサービスが起動しない場合、SystemログにService Control Managerのイベントが記録されることがあります。 起動タイムアウト、依存サービス未起動、実行ファイル不在、ログオン失敗、権限不足などを疑います。

1. アプリ名またはサービス名を確認
利用者報告、監視、タスクマネージャー、サービス管理画面から対象を特定します。
2. 発生時刻のApplicationログを見る
クラッシュ、.NET例外、Windows Error Reportingの有無を確認します。
3. Systemログを見る
サービス起動失敗、依存関係、ドライバ、ディスク、ネットワークの異常を確認します。
4. 実行条件を確認
実行ユーザー、ファイル権限、DLL、設定ファイル、ポート、証明書、レジストリを確認します。
よくある落とし穴： クラッシュログに出た障害モジュール名だけで原因を決めないでください。依存DLL、入力データ、権限、設定、直前の更新が引き金になっている場合があります。

6. Applications and Services Logs

Applications and Services Logsには、Windows機能やアプリケーションごとの詳細ログが含まれます。 標準のApplicationやSystemだけでは情報が足りない場合、ここに詳細なイベントが記録されていることがあります。

ログ例 見る場面
Microsoft-Windows-TaskScheduler タスクスケジューラの起動、終了コード、失敗理由を確認する。
Windows PowerShell / PowerShellCore PowerShell実行、スクリプト、リモート管理の調査に使う。
Microsoft-Windows-Windows Defender 検出、隔離、保護機能、スキャン結果を確認する。
Microsoft-Windows-DNS-Client 名前解決やDNSクライアント周辺の調査に使う。
IIS関連ログ IIS管理、アプリケーションプール、HTTP.sysなどの詳細調査で見る。

ログによっては既定で無効、または詳細ログが別途有効化されていない場合があります。 本番環境で詳細ログを増やす場合は、ディスク容量、個人情報、監査要件、性能影響を確認します。

7. 障害時の最初の確認ポイント

障害時は、ログを読む前に調査範囲を決めます。 時刻、対象端末、対象ユーザー、対象アプリ、直前の変更、再現条件を確認してからイベントログを見ると、不要なログに振り回されにくくなります。

1. 何が起きているか
エラー画面、業務影響、利用者の操作、再現条件を確認します。
2. いつから起きているか
発生時刻、最終正常時刻、更新やリリースの時刻を確認します。
3. どこで起きているか
全端末か一部端末か、特定ユーザーか、特定サーバーかを確認します。
4. どのログを見るか
Application、System、Security、Setup、アプリ固有ログを選びます。
5. 仮説を分ける
アプリ、権限、サービス、ネットワーク、証明書、更新、設定のどれが濃いかを整理します。
実務の型： ログを見る前に、発生時刻と対象を絞ることが重要です。ログは多いため、先に調査範囲を決めます。

8. 実務での観点

イベントログは、障害対応、監査、運用改善、再発防止に使えます。 ただし、イベントログだけでは業務処理の詳細やアプリ内部状態までは分からないことがあります。 そのため、アプリログ、Webサーバーログ、DBログ、監視ログ、リリース記録と組み合わせて読みます。

  • 利用者報告の時刻と、イベントログの時刻を合わせて確認します。
  • エラーだけでなく、直前の警告や情報ログも確認します。
  • 同じイベントが繰り返されているか、単発かを確認します。
  • サービス停止や再起動は、Systemログとサービス固有ログを合わせて見ます。
  • 認証や権限の問題は、Securityログを見る必要がある場合があります。
  • ログ保持期間、最大サイズ、上書き設定を確認し、必要なログが消えないようにします。
運用上の注意： Securityログや監査ログには機密性の高い情報が含まれる場合があります。収集、共有、保管は組織のルールに従います。

9. コマンド例

イベントログは、PowerShellで取得すると、条件指定、絞り込み、CSV出力、自動収集がしやすくなります。 現代のWindowsでは、Get-WinEvent を優先して覚える価値があります。

目的 コマンド例 確認できること
Systemログの直近確認 Get-WinEvent -LogName System -MaxEvents 20 Systemログの最新イベントを確認できます。
Applicationログのエラー抽出 Get-WinEvent -FilterHashtable @{LogName='Application'; Level=2} Applicationログのエラーイベントを抽出できます。
期間指定 Get-WinEvent -FilterHashtable @{LogName='System'; StartTime=(Get-Date).AddHours(-2)} 直近2時間のSystemログを確認できます。
サービス関連ログ Get-WinEvent -LogName System | Where-Object ProviderName -eq 'Service Control Manager' サービス起動・停止・失敗に関係するイベントを確認できます。
CSV出力 Get-WinEvent -LogName Application -MaxEvents 100 | Export-Csv events.csv 調査結果をCSVとして保存し、共有や比較に使えます。 
Get-WinEvent -LogName Application -MaxEvents 50 |
  Select-Object TimeCreated, ProviderName, Id, LevelDisplayName, Message

Get-WinEvent -FilterHashtable @{
  LogName = 'System'
  ProviderName = 'Service Control Manager'
  StartTime = (Get-Date).AddHours(-6)
}

Get-WinEvent -FilterHashtable @{
  LogName = 'Application'
  Level = 2
  StartTime = (Get-Date).AddDays(-1)
} | Export-Csv .\application-errors.csv -NoTypeInformation -Encoding UTF8
使い分け： GUIで概要を確認し、PowerShellで条件を固定して再現性のあるログ取得を行うと、調査記録として残しやすくなります。

第8章のまとめ

  1. イベントログは、Windowsやアプリが記録する事象のログであり、障害調査の一次情報になる。
  2. イベントビューアーでは、発生時刻、ログ名、ソース、イベントID、レベル、メッセージをセットで見る。
  3. Applicationはアプリ、SystemはOSやサービス、Securityは認証や監査、Setupは更新やインストールの調査で使う。
  4. エラー、警告、情報は重要度が異なるが、エラーだけを見ればよいわけではない。
  5. イベントIDはソースとセットで解釈し、番号だけで原因を決めない。
  6. クラッシュ調査では、Applicationログとアプリログ、障害モジュール、実行条件を合わせて確認する。
  7. サービス起動失敗では、Systemログ、Service Control Manager、実行ユーザー、依存関係、権限を確認する。
  8. Get-WinEventを使うと、期間指定、レベル指定、CSV出力などの再現性ある調査ができる。

10. 理解度チェック

Q1. イベントログは障害調査でどのような役割を持ちますか。

回答例を見る

障害発生時刻に、Windows、サービス、アプリ、認証、更新などで何が記録されたかを確認するための一次情報です。 原因を断定するものではなく、仮説を立てる材料として使います。

Q2. ApplicationログとSystemログの違いは何ですか。

回答例を見る

Applicationログはアプリケーションやランタイムのイベントを中心に記録します。 SystemログはOS、サービス、ドライバ、デバイスなどシステム側のイベントを中心に記録します。

Q3. イベントIDだけで原因を判断してはいけない理由は何ですか。

回答例を見る

イベントIDはソースごとに意味が異なるためです。 ログ名、ソース、イベントID、メッセージ、発生時刻、環境をセットで確認する必要があります。

Q4. Windowsサービスが起動しない場合、どのログを確認しますか。

回答例を見る

まずSystemログでService Control Managerのイベントを確認します。 併せてApplicationログ、アプリ固有ログ、サービス実行ユーザー、依存サービス、権限、設定ファイルを確認します。

Q5. Get-WinEventを使う利点は何ですか。

回答例を見る

ログ名、期間、レベル、ソースなどで条件指定でき、結果をCSV出力したり、複数端末の調査や定期収集に使いやすい点です。
