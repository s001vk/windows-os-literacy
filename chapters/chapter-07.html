Windows OS教養講座 第7章

レジストリ

レジストリを、OSやアプリケーションの設定を保存する基盤として理解し、グループポリシー、権限、障害調査との関係を学びます。

この章の目的

第6章では、Windowsのネットワーク基礎を扱いました。 第7章では、Windowsの動作やアプリケーション設定に強く関係するレジストリを学びます。

レジストリは、OS設定、アプリ設定、サービス設定、関連付け、ポリシー、インストール情報などを保存する階層型の設定データベースです。 業務システム運用では、設定ファイルだけでなく、レジストリに保存された値が動作を左右することがあります。

この章のゴール： レジストリを安易に編集する対象ではなく、設定、権限、ポリシー、障害調査の観点で安全に確認できるようになることです。

1. レジストリとは何か

レジストリは、Windowsが利用する設定データベースです。 ファイルシステムのように階層構造を持ち、キーと値で設定を管理します。 設定ファイルがアプリごとに分かれているのに対し、レジストリはOSやアプリが共通して参照する設定の保存場所として使われます。

レジストリには、OSの構成、デバイス、サービス、ユーザーごとの設定、アプリケーションのインストール情報、ファイル拡張子の関連付け、ポリシー設定などが保存されます。 そのため、Windowsの挙動を調査するときに、ファイル、プロセス、サービス、ログと並んで確認対象になります。

キーフォルダのような階層
値設定名とデータ
ハイブ大きな管理単位
権限読み取り、変更、所有者
注意： レジストリは重要な設定基盤です。意味を理解しないまま変更すると、アプリやOSの動作に影響する場合があります。

2. 主要なハイブ

レジストリの最上位に見える分類を、ハイブと呼びます。 実務では特に HKEY_LOCAL_MACHINE と HKEY_CURRENT_USER をよく確認します。

ハイブ 主な意味 実務で見る場面
HKEY_LOCAL_MACHINE 端末やサーバー全体に関係する設定です。 サービス、ドライバ、インストール済みアプリ、OS全体設定、マシン単位ポリシー。
HKEY_CURRENT_USER 現在ログオンしているユーザーの設定です。 ユーザー別アプリ設定、画面設定、ユーザー単位ポリシー、資格情報に近い設定。
HKEY_CLASSES_ROOT ファイル拡張子、COM、関連付けなどの情報です。 ファイルを開くアプリ、COMコンポーネント、Office連携、古い業務アプリ。
HKEY_USERS 各ユーザープロファイルの設定をSID単位で保持します。 ログオンしていないユーザーの設定確認、サービスアカウントのユーザー設定調査。
HKEY_CURRENT_CONFIG 現在のハードウェア構成に関する情報です。 通常の業務アプリ調査では頻度は低いが、デバイス関連調査で見ることがあります。
基本の見方： マシン全体の設定はHKLM、ユーザーごとの設定はHKCUとして切り分けると、手動実行とサービス実行の違いを理解しやすくなります。

3. アプリ設定とOS設定

Windowsアプリは、設定ファイルだけでなくレジストリに設定を保存することがあります。 特に、インストーラーで導入するデスクトップアプリ、古い業務アプリ、COMを使うアプリ、ドライバやサービスを伴うアプリでは、レジストリが重要になります。

OS側でも、サービス定義、起動設定、デバイス、ネットワーク、セキュリティ、ファイル関連付けなど、多くの情報がレジストリに保存されます。 ただし、すべての設定をレジストリで管理しているわけではなく、設定ファイル、管理DB、クラウドポリシー、グループポリシーも関係します。

アプリ本体EXE、DLL、ランタイム
設定ファイルJSON、XML、INI、config
レジストリインストール情報、関連付け、ポリシー
ログイベントログ、アプリログ
調査の考え方： アプリの設定値が見つからない場合、設定ファイル、環境変数、レジストリ、グループポリシー、アプリ管理画面を分けて確認します。

4. グループポリシーとの関係

企業環境では、グループポリシーによってWindows端末やユーザーの設定を一括管理します。 グループポリシーで配布された設定の一部は、レジストリに反映されます。 そのため、レジストリ上の値を手で変更しても、次回のポリシー適用で戻ることがあります。

ポリシーが関係する設定は、利用者やアプリから見ると「なぜか変更できない」「再起動すると戻る」「特定端末だけ違う」ように見える場合があります。 その場合、レジストリ単体ではなく、グループポリシーの適用結果も確認します。

1. 管理者がポリシーを定義
Active Directoryや端末管理基盤で、ユーザーや端末に適用する設定を決めます。
2. 端末へポリシーが適用
ログオン時、起動時、定期更新、手動更新などで設定が反映されます。
3. レジストリに反映
一部のポリシーはレジストリ値として保存され、OSやアプリが参照します。
4. アプリやOSの動作が変わる
プロキシ、セキュリティ設定、更新設定、ブラウザ設定などに影響します。
注意： 管理された環境でポリシー由来のレジストリ値を手動変更しても、恒久対応にならない場合があります。原因がポリシーなら、管理元での修正が必要です。

5. レジストリの権限

レジストリにもファイルシステムと同じように権限があります。 キーごとに、読み取り、値の設定、サブキー作成、所有者、監査などを制御できます。 権限不足により、アプリが設定を読めない、書き込めない、サービス起動時だけ失敗することがあります。

観点 確認内容 実務での例
読み取り権限 アプリやサービスが設定値を読めるか。 サービスアカウントでは設定参照に失敗する。
書き込み権限 設定値やサブキーを作成・変更できるか。 初回起動時の設定保存やライセンス情報保存に失敗する。
ユーザー差異 HKCUの参照先が実行ユーザーごとに違う。 手動実行では動くが、タスクスケジューラやサービスでは動かない。
32bit / 64bit差異 32bitアプリが別のレジストリビューを参照する場合がある。 本番だけCOM登録やODBC設定が見つからない。
第4章との接続： レジストリ調査でも、実行ユーザー、グループ、管理者権限、UAC、サービスアカウントを分けて考える必要があります。

6. 直接編集の危険性とバックアップ

レジストリは、レジストリエディターやコマンドで編集できます。 しかし、直接編集は影響範囲が大きく、間違えるとアプリが起動しない、サービスが動かない、ログオンやOS設定に影響する場合があります。

変更が必要な場合は、変更理由、対象キー、変更前の値、変更後の値、戻し方を記録します。 可能であれば、アプリの管理画面、インストーラー、グループポリシー、PowerShellの管理コマンドなど、正式な設定手段を優先します。

1. 変更理由を明確にする
何の不具合を解消するために、どの設定を変えるのかを記録します。
2. 対象範囲を限定する
HKLMかHKCUか、端末全体かユーザー単位か、32bit / 64bit差異があるかを確認します。
3. 変更前にエクスポートする
対象キーをバックアップし、戻せる状態にします。
4. 変更後に動作確認する
アプリ、サービス、イベントログ、ポリシー再適用後の状態を確認します。
安全な原則： 調査目的で広範囲のキーを削除したり、出典不明の.regファイルを取り込んだりしないでください。企業端末では変更管理の対象として扱います。

7. 原因調査で見る場面

レジストリは、常に最初に見る対象ではありません。 まずは、エラーメッセージ、アプリログ、イベントログ、実行ユーザー、ファイル、通信、サービス状態を確認します。 その上で、設定値、関連付け、ポリシー、インストール状態が疑われる場合にレジストリを確認します。

症状 レジストリで見る候補 合わせて見るもの
アプリが起動しない インストール情報、COM登録、ランタイム設定、関連付け。 イベントログ、DLL、実行ファイル、権限、ランタイム。
ユーザーごとに挙動が違う HKCU、HKEY_USERS、ユーザー単位ポリシー。 ユーザープロファイル、グループ、AppData、ポリシー適用結果。
サービスが起動しない サービス定義、ImagePath、依存関係、実行アカウント周辺設定。 サービス管理、イベントログ、実行ファイル、権限。
設定が戻る Policies配下の値、管理テンプレート由来の設定。 gpresult、グループポリシー、Intuneなどの管理基盤。
32bitアプリだけ失敗する WOW6432Node、32bit側のCOMやODBC設定。 プロセスのbit数、DLL、ODBC管理ツール、ランタイム。

8. 実務での観点

業務システム運用では、レジストリは「変更する場所」よりも「設定の由来を確認する場所」として扱うことが多くなります。 変更する場合も、手作業ではなく、インストーラー、管理コマンド、グループポリシー、構成管理ツールで再現可能にすることが重要です。

  • 同じ設定でも、HKLMとHKCUのどちらにあるかで影響範囲が変わります。
  • サービスやIISでは、対話ユーザーのHKCUではなく、実行アカウントの設定が関係します。
  • グループポリシー由来の設定は、手動変更しても戻る場合があります。
  • 32bit / 64bit差異により、見ているレジストリビューが違う場合があります。
  • 変更時は、対象キー、変更前後の値、実施者、日時、戻し手順を記録します。
運用の型： レジストリは、ファイル、権限、プロセス、サービス、通信、ログと組み合わせて確認すると、設定起因の問題を切り分けやすくなります。

9. コマンド例

レジストリはGUIのレジストリエディターだけでなく、cmdやPowerShellでも確認できます。 コマンドで確認すると、調査結果を記録しやすく、複数端末の比較にも使えます。

目的 コマンド例 確認できること
キーの確認 reg query "HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion" 指定キー配下の値やサブキーを確認できます。
値の取得 Get-ItemProperty "HKCU:\Software\Microsoft\Windows\CurrentVersion\Explorer" PowerShellでレジストリ値をオブジェクトとして確認できます。
対象キーのバックアップ reg export "HKCU\Software\Example" backup.reg 変更前に対象キーを.regファイルへエクスポートできます。
サービス定義確認 reg query "HKLM\SYSTEM\CurrentControlSet\Services\サービス名" サービスのImagePathや依存関係に近い情報を確認できます。
ポリシー適用結果 gpresult /r レジストリ値の背景にあるグループポリシー適用状況を確認できます。 
reg query "HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall"
reg export "HKCU\Software\ExampleApp" "%USERPROFILE%\Desktop\ExampleApp-backup.reg"
Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion"
Get-ChildItem "HKLM:\SYSTEM\CurrentControlSet\Services" | Select-Object -First 10
変更コマンドの注意： reg add や reg delete は実際に設定を変更します。本番環境では、事前バックアップ、影響確認、戻し手順、変更記録を用意してから実行します。

第7章のまとめ

  1. レジストリは、Windowsやアプリケーションが利用する階層型の設定データベースである。
  2. HKLMはマシン全体、HKCUは現在ユーザーの設定として理解すると切り分けやすい。
  3. アプリ設定は、設定ファイル、環境変数、レジストリ、ポリシーなど複数の場所に分かれることがある。
  4. グループポリシーで配布された設定は、レジストリに反映され、手動変更しても戻る場合がある。
  5. レジストリにも権限があり、実行ユーザーやサービスアカウントによって読み書き可否が変わる。
  6. 32bit / 64bit差異により、参照するレジストリビューが異なる場合がある。
  7. 直接編集する場合は、対象キーのバックアップ、変更前後の記録、戻し手順が必要である。
  8. レジストリ調査は、ログ、ファイル、権限、プロセス、サービス、通信と合わせて行う。

10. 理解度チェック

Q1. レジストリとは何ですか。

回答例を見る

Windowsやアプリケーションが利用する階層型の設定データベースです。 OS設定、アプリ設定、サービス、関連付け、ポリシー、インストール情報などが保存されます。

Q2. HKLMとHKCUの違いは何ですか。

回答例を見る

HKLMは端末やサーバー全体に関係するマシン単位の設定です。 HKCUは現在ログオンしているユーザーの設定です。

Q3. グループポリシーとレジストリはどのように関係しますか。

回答例を見る

グループポリシーで配布された設定の一部は、レジストリ値として端末やユーザーに反映されます。 手動で変更しても、次回のポリシー適用で戻る場合があります。

Q4. レジストリを直接編集する前に確認すべきことは何ですか。

回答例を見る

変更理由、対象キー、影響範囲、変更前の値、バックアップ、戻し手順、変更後の確認方法、変更記録を確認します。

Q5. 手動実行では動くがサービスでは動かない場合、レジストリ面で何を疑いますか。

回答例を見る

サービス実行ユーザーが異なるためHKCUの参照先が違う、レジストリキーの読み取り権限がない、32bit / 64bitのレジストリビューが違う、ポリシー適用状態が違う、といった点を疑います。
